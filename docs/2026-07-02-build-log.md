## Networking: What Actually Worked

All three VMs, Security Onion's sniffing adapter, the Windows Server victim, and Kali, are joined to the same VirtualBox NAT Network (created via File > Tools > Network Manager). This gives every VM real internet access, and because it's a shared virtual switch rather than isolated per-VM NAT, Security Onion's sniffing adapter can see all traffic on the segment when set to promiscuous mode.

Security Onion runs two adapters:
- Adapter 1 (management): Bridged to host, used for reaching the SOC web console
- Adapter 2 (sniffing): NAT Network, Promiscuous Mode set to Allow All

Windows Server and Kali each run a single adapter on the same NAT Network.

## Networking: Dead Ends Along the Way

Worth documenting since these are common mistakes and the reasoning behind why they don't work:

- **VirtualBox Internal Network mode cannot include a physical machine.** This came up early when Kali was planned as a physical box. Internal Network exists entirely inside VirtualBox's software layer and has no path to any physical NIC.
- **Plain NAT isolates each VM individually.** Windows Server on regular NAT and Security Onion on a separate NAT adapter cannot see each other's traffic, since each VM gets its own private NAT instance. NAT Network (a distinct VirtualBox feature) is what allows multiple VMs to share one virtual switch with shared internet access.
- **Bridged mode with promiscuous mode does work for capturing internet-bound traffic**, but requires the host's physical Ethernet port to actually have an internet uplink (via a switch or ICS), which added hardware/config complexity that became unnecessary once Kali was converted from physical to a VM.
- **Kali was originally planned as a physical machine**, which drove most of the bridging/switch complexity. Checking actual host RAM headroom (32GB total, ~20GB free with Security Onion and the victim VM running) confirmed there was room to run Kali as a third VM instead, which simplified the entire network design to the NAT Network approach above.

## Security Onion: bond0

Security Onion automatically enslaves the assigned monitor interface (`enp0s8` in this build) into a bonded interface (`bond0`), and Suricata/Zeek listen on `bond0`, not the raw physical adapter. This is standard Security Onion architecture, not a misconfiguration. Confirmed via:

```
sudo salt-call pillar.get sensor:interface
```

which correctly returned `bond0`.

## Verification Steps Performed

1. `tcpdump -i enp0s8` confirmed raw packets hitting the physical sniffing adapter (NTP traffic observed first, confirming the interface is live).
2. `tcpdump -i bond0` confirmed traffic also visible at the bond layer, confirming the full capture chain from wire to bond.
3. Generated live traffic from the Windows Server VM (HTTPS to an external IP) and confirmed it was visible in tcpdump on both interfaces.
4. Confirmed the same traffic surfaced in the SOC web console under Dashboards > Connections seen by Zeek or Suricata.

## Guest Additions Notes

- Windows Server and Kali: standard install via Devices > Insert Guest Additions CD image, worked as expected.
- Security Onion: intentionally **not installed**. Security Onion's OS (Oracle Linux 9 with the UEK kernel) doesn't have official support for VirtualBox Guest Additions, the install attempt failed on a kernel header mismatch (UEK headers vs. generic RHEL headers), and Guest Additions provides no functional benefit here since Security Onion is managed entirely through its web console, not the VM's console window.
