## Reconnaissance: Initial Nmap Scan Was Wrong

First `nmap -A 10.0.2.3` from Kali did not actually reach the victim. It returned a MAC address of `52:54:00:12:35:00`, which is a QEMU virtual NIC OUI, and only port 53 open running dnsmasq. That fingerprint matches VirtualBox's own internal NAT Network DHCP and DNS service, not a Windows machine.

Root cause: `VBoxManage showvminfo "Attacker-Kali" --machinereadable` showed `nic1="nat"` and `nic2="intnet"`. Kali had never actually joined the shared NAT Network, it was still on plain NAT plus an Internal Network adapter, neither of which can reach the victim VM. It had likely reset to defaults at some point after the network was originally configured.

Fixed with:

```
VBoxManage modifyvm "Attacker-Kali" --nic1 natnetwork --nat-network1 "<network name>"
VBoxManage modifyvm "Attacker-Kali" --nic2 none
```

After the fix, `nmap -A 10.0.2.3` returned the correct MAC (`08:00:27:D4:76:EC`, VirtualBox's own OUI) and the real open ports:

```
5357/tcp open  http    Microsoft HTTPAPI httpd 2.0 (SSDP/UPnP)
5985/tcp open  http    Microsoft HTTPAPI httpd 2.0 (SSDP/UPnP)
```

OS fingerprint guessed Windows Server 2019 at 92% confidence.

## Port 5985: WinRM

Port 5985 is the default HTTP listener for Windows Remote Management (WinRM), used for remote PowerShell sessions and remote command execution. This became the target service for the brute force scenario. Follow-up scan confirmed the service directly:

```
nmap -p 5985 --script http-title,http-headers,http-methods 10.0.2.3
```

Returned `5985/tcp open wsman`, confirming WinRM specifically rather than a generic HTTP service.

## Kali DNS Failure

Separately from the NAT Network fix, Kali lost internet access for `apt` and the browser while ping still worked intermittently. `cat /etc/resolv.conf` showed the primary and secondary DNS servers as `198.18.0.1` and `198.18.0.2`, which is an IANA reserved range for network benchmark testing, not real DNS servers. Linux's libc resolver only tries the first three nameservers listed, so Kali never reached the working fallback further down the list, while Windows Server (not limited to three) did.

Suspected but unconfirmed cause: a VPN active on the host machine around this time, which may have altered the host's DNS and gotten passed through to guests via the NAT Network's DHCP.

Fixed by overriding DNS on the connection profile so it survives DHCP renewals:

```
sudo nmcli connection modify "Wired connection 1" ipv4.dns "1.1.1.1 8.8.8.8"
sudo nmcli connection modify "Wired connection 1" ipv4.ignore-auto-dns yes
sudo nmcli connection down "Wired connection 1"
sudo nmcli connection up "Wired connection 1"
```

## Password Variant Generator

Built a tool to expand a base wordlist into brute-forceable variants for the T1110.002 mask attack scenario, since running a raw wordlist like rockyou.txt directly against WinRM was too slow.

Transformations implemented, each behind its own CLI flag:

- `-u` uppercase the first letter if lowercase
- `-l` leet substitutions (a to @/4, e to 3, i to 1/!, o to 0, u to v, s to 5)
- `-e` append `!`
- `-y` append 2-digit and 4-digit years, 1950 through the current year (pulled dynamically via `datetime`), each followed by `!`
- `-w<N>` wrap the password with a special character plus a single specified digit `N` plus the same special character, across 14 common special characters

Only 1 or 2 flags allowed per run, space-delimited, no shorthand combining, duplicate flags rejected. Output contains only newly generated variants, not the original input passwords.

First version was written in Python. It worked on small inputs but repeatedly hit the Kali VM's 8GB RAM ceiling and got killed by the OOM killer when run against rockyou.txt (roughly 14 million entries) or even a 100,000-entry subset with the `-l` flag, since leet substitution is combinatorial and the in-memory deduplication set grew past available RAM.

Rewrote the tool in Rust using `rayon` for multi-threaded processing across the VM's 4 cores, with `clap` for flag parsing. Compiled in release mode for optimizations. The Rust version is faster per operation but still hit the same fundamental memory ceiling, since both versions hold a deduplication set in RAM rather than deduplicating on disk. Running `-l` alone against a 100,000-entry input still got killed.

Working configuration: a 10,000-entry subset of rockyou.txt (`head -n 10000`) with `-e -l` completed successfully, producing the final wordlist used in the attack, `maskedtest.txt`. A follow-up fix to remove in-memory dedup in favor of an external `sort -u` pass was identified but not yet implemented, noted for a future session if larger inputs are needed.

## T1110.002: Mask Attack via NetExec

Installed NetExec (`nxc`), the maintained successor to CrackMapExec, since the initial flag guesses (`-U`/`-P`) were wrong and had to be corrected against the actual `nxc winrm --help` output.

Ran the brute force against the victim's WinRM service using a username wordlist (SecLists `malenames-usa-top1000.txt`) against the generated password wordlist (`maskedtest.txt`):

```
nxc winrm 10.0.2.3 -u <username wordlist> -p <maskedtest.txt>
```

Confirmed along the way that Windows usernames are case insensitive at the OS level, so the all-caps names in the SecLists file authenticate the same as their lowercase equivalents. Only the password side is case sensitive.
